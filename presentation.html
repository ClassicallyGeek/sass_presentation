Why Sass and moreso why SCSS

PATTY

Sass

  • Indented syntax of Sass was inherited from HAML
  • Sass doesn't complain about missing semi-colons
PATTY sass syntax uses indentation rather than brackets to indicate nesting of selectors, and newlines rather than semicolons to separate properties. more syntactically concise.

EXAMPLE

						
=border-radius($radius)
	-webkit-border-radius: $radius
	-moz-border-radius:    $radius
	-ms-border-radius:     $radius
	border-radius:         $radius

	.box
	+border-radius(10px)
PATTY

SCSS

  • Superset of CSS. Contains all the features of CSS and features of Sass as well
  • encourages proper nesting of rules
  • encourages more modular code with @extend
  • existing tools often work with SCSS
  • integration with an existing CSS codebase is much easier
  • SCSS provides a much lower barrier to entry
PATTY

EXAMPLE

						
@mixin border-radius($radius) {
  -webkit-border-radius: $radius;
     -moz-border-radius: $radius;
      -ms-border-radius: $radius;
          border-radius: $radius;
}

.box { @include border-radius(10px); }

Leverage imports and partials

to create a modularized code base for maintainability
PATTY
With traditional CSS, modularisation is implemented through classes and ids but Sass allows us to take this idea one step further through the use of functions and partials.
  • The idea here is that each module/action within the app has their own Sass file.
  • The idea is that anything that can made modular should be poosed int a partial so it can be reused anywhere.
  • Inside which only partials and mixins needed for taht module will be imported. Similar to a well structured webapp where only the required functions and classes are imported so performance is optimised.

PATTY
PARTIALS:
  • you can create partial Sass files that contain little snippets of CSS taht you can include in other Sass files.
  • This is a great way to modularize your CSS and help keep thigns easier to maintain. A partial is simply a Sass file named with a leading underscore.
  • The underscore lets Sass know that the file is only a partial file and that it should not be generated into a CSS file.
  • Sass partials are used with the @import directive.
PATTY

@Import

						
@import "mixins/colors";
@import "mixins/typography";
@import "mixins/functions";
@import "partials/reset";
@import "partials/base";
@import "partials/layout";
@import "partials/buttons";
PATTY
IMPORT: CSS has an import option that lets you split your CSS into smaller, more maintainable portions. The only drawback is that each time you use @imporrt in CSS it creates another HTTP request. Sass bulds on top of the current CSS @import but instead of requiring an HTTP request, Sass will take the file you want to import and combine it with the file you're importing into so you can serve a single CSS file to the web broweser.

The Compass Lifestyle

  • create (create a new compass project)
  • init (initialize an existing project to work with compass)
  • install (install a pattern from an extension into a project)
  • compile (compile the project's sass files into CSS)
  • watch (watch the project for changes and compile whenever it does)
  • config (emit a configuration file at the location specified)
  • validate (validate the generated CSS)

Mixins always compile to CSS rules where they appear in code. Where an @extend generates more selectors, @mixin always generates more rules. This is why for static rules it is often (not always) less bloaty to use @extend.
As a rule of thumb, I only use mixins to do "mathy" things. "If it doesn't need to pass variable, it doesn't need to be a mixin" is my motto

Mixins

PATTY

Example

						
@mixin down-arrow($color, $size) {
	content: ' ';
	border-style: solid;
	border-width: $size $size 0 $size;
	border-color: $color transparent transparent transparent;
}

@mixin up-arrow($color, $size) {
	content: ' ';
	border-style: solid;
	border-width: 0 $size $size $size;
	border-color: transparent transparent $color transparent;
}

Mixins vs Extends

While an @extend generates more selectors, @mixin always generates more rules.
PATTY

@extend

						
.boom {
	color: blue;
}

.pow {
	@extend boom;
}
CSS output: 
						
. boom, .pow {
	color: blue;
}
PATTY Notice: it doesn't "fetch" the style from .boom, instead it commma separates the original selector. It applies the styles to both more efficeintly.

@mixin

						

@mixin test {
  color: red;
}
.test1 {
  @include test;
} 
.test2 {
  @include test;
}
CSS output 
					
.test1 {
  color: red;
}
.test2 {
  color: red;
}
